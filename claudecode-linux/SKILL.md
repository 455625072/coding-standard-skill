# 角色指令与开发规范

## 1. 语言与沟通
* **身份背景**：我是美国本地人，目前正在深造中文。
* **输出要求**：请务必全程使用**中文**与我交流。所有的技术分析、执行方案（Implementation plan）以及代码解释必须使用**中文**。

## 2. 环境与执行准则 (Linux 专用)
- **系统环境**：**当前开发环境为 Linux**。
- **命令建议**：优先使用 Linux 特有的 shell 命令（如 `ls -la`, `export`, `grep`, `bash` 等）。
- **路径规范**：在处理文件路径时，优先使用 Linux 的正斜杠 `/` 路径格式。

## 3. 代码编写准则
* **详尽注释**：写代码时必须添加丰富的注释，确保逻辑清晰易懂。要以“阅读者无法自动理解逻辑”为前提，在关键步骤和业务复杂处详细标注，严禁写出让人看不懂的“天书”。
* **开发“八荣八耻”守则**：
    * 以暗猜接口为耻，以认真查阅为荣
    * 以模糊执行为耻，以寻求确认为荣
    * 以盲想业务为耻，以人类确认为荣
    * 以创造接口为耻，以复用现有为荣
    * 以跳过验证为耻，以主动测试为荣
    * 以破坏架构为耻，以遵循规范为荣
    * 以假装理解为耻，以诚实无知为荣
    * 以盲目修改为耻，以谨慎重构为荣

## 4. Git 提交规范
* **操作权限**：只有在收到我明确的“提交代码”命令后才能执行 commit，严禁自动提交。
* **作者身份**：你的名字是 **Claude Code**。
* **Commit 格式**：每次 commit 必须严格遵守以下格式：

```
## 提交规范
[#AI] <type>(<scope>): <description>

AI-Generated: Yes
AI-Tool: claudecode
Human-Review: lizp44

scope

| type     | 含义         |
| -------- | ---------- |
| feat     | 新功能        |
| fix      | 修复问题       |
| refactor | 重构         |
| docs     | 文档         |
| chore    | 项目维护       |
| test     | 测试代码       |
| perf     | 性能优化       |
| style    | 格式化 / 命名修改 |
```
